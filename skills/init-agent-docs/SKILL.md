# 初始化 Agent 文档体系

> 为项目创建 AI Agent 可理解的文档体系，以 AGENTS.md 为入口，docs/ 按需索引深度文档。

## 核心原则

- **README.md 面向人类开发者** -- 用于安装、初始化、构建、运行、测试等上手流程，不承担 Agent 规则入口职责
- **AGENTS.md 是唯一入口** -- 所有 Agent 自动加载的规则从这里开始
- **CLAUDE.md 只做轻量补充** -- 通过 `@AGENTS.md` 引用主规则，仅添加 Claude 专属指令
- **docs/ 按需加载** -- 详细文档不放入口文件，Agent 根据任务主动读取
- **docs/ 描述"是什么"，skills/ 描述"怎么做"** -- 避免重复

## 目录结构

```
repo/
├─ README.md              # 面向开发者的安装/初始化/构建说明（建议 ~40 行）
├─ AGENTS.md              # 入口导航（~80 行）
├─ CLAUDE.md              # Claude 补充（~10 行，@AGENTS.md）
├─ docs/
│  ├─ architecture/       # 系统架构（技术栈、分层、核心模块关系）
│  ├─ domains/            # 业务模型（实体、领域概念、数据流）
│  ├─ conventions/        # 代码规则（命名、格式、模式约定）
│  └─ workflows/          # 开发流程（构建、测试、发布、Git）
├─ apps/
│  └─ <app>/
│     ├─ AGENTS.md        # 应用级规则 + 导航
│     └─ docs/            # 应用级详细文档（结构同上）
```

**多应用/多模块项目**：每个独立部署的应用或可独立开发的模块应有自己的 `AGENTS.md`，确保独立开发时文档自足。

## README.md 写法

README.md 面向人类开发者，用于帮助新同学或首次接手项目的人快速完成安装、初始化、运行、构建、测试。它不是 Agent 规则入口，不要在 README.md 中堆放大量 Agent 指令。

### README.md 应包含什么

- **项目简介** -- 一句话说明项目用途、技术栈、运行平台
- **环境要求** -- 语言/SDK 版本、包管理器、系统依赖、网络条件
- **安装与初始化** -- clone、安装依赖、生成配置、初始化数据库/证书/子模块等
- **本地开发** -- 启动命令、工作区/入口文件、常见开发方式
- **构建 / 测试** -- 最常用的构建、测试、lint 命令
- **常见说明** -- 私有依赖、权限要求、首次运行常见问题
- **相关文档** -- 指向 AGENTS.md 和 docs/ 深入文档

### README.md 不应承担什么

- 不作为 Agent 自动加载入口
- 不堆放大量架构细节，架构细节应放到 `docs/architecture/`
- 不复制 AGENTS.md 的规则列表，避免双重维护

### 初始化原则

- 如果项目存在明确初始化步骤（如 `make init`、`cp .env.example .env`、`git submodule update --init --recursive`、证书下载、数据库迁移），README.md 必须写清楚顺序
- 如果项目没有额外初始化步骤，明确写出“安装依赖后即可启动”，避免让读者猜
- 命令优先按“首次接手项目”的执行顺序组织，而不是按概念分组

## CLAUDE.md 写法

CLAUDE.md 仅包含 Claude Code 专属指令，通过 `@AGENTS.md` 引用主规则：

```markdown
# 项目名

@AGENTS.md

## Claude 专用

- compact 时保留已修改文件列表和当前任务上下文
- 使用 subagent 探索代码库，保持主上下文干净
- 单模块测试优先：`<测试单模块的命令>`，避免跑全量
```

**要点**：

- 总行数控制在 10-20 行
- `@AGENTS.md` 必须在文件顶部，确保 Agent 自动加载主规则
- 只放 Claude Code 特有的行为指令（compact 策略、subagent 用法等）
- 通用规则一律写在 AGENTS.md

## AGENTS.md 写法

AGENTS.md 是 Agent 自动加载的入口文件，包含项目全局规则和文档索引。

### 必选节

| 节名         | 内容                                             |
| ------------ | ------------------------------------------------ |
| **命令**     | 常用命令（安装、构建、开发、测试），用反引号包裹 |
| **架构**     | 项目结构概述、模块关系、技术栈，3-5 条           |
| **代码风格** | 语言、格式化、命名规范，5-8 条                   |
| **Git**      | 提交格式、分支策略，3-5 条                       |
| **边界**     | 禁止修改的文件/目录、跨模块限制                  |
| **文档索引** | 指向 docs/ 下各文档的路径和一句话说明            |

### 可选节（按项目需要添加）

- **依赖管理** -- 包管理器约定、版本策略
- **测试** -- 测试框架、覆盖率要求
- **部署** -- 环境、CI/CD 相关规则

### 行数控制

- 根目录 AGENTS.md：**80 行以内**
- 应用级 AGENTS.md：**100 行以内**
- 超出则拆分到 docs/

### 示例结构

```markdown
# 项目名

## 命令

- `<安装命令>` -- 安装依赖
- `<构建命令>` -- 构建项目
- `<开发命令>` -- 启动开发服务器
- `<测试命令>` -- 运行测试

## 架构

- `src/` -- 源代码目录，按功能模块组织
- `packages/` -- 共享库
- 架构详见 `docs/architecture/<核心架构文档>.md`

## 代码风格

- 使用 TypeScript
- 优先使用命名导出
- 2 空格缩进
- 格式化工具：Prettier

## Git

- 提交格式：`type(scope): description`
- type 类型：`feat`, `fix`, `refactor`, `docs`, `test`, `chore`

## 边界

- `dist/`、`node_modules/`、`.env*` -- 不要修改
- lock 文件 -- 不要手动编辑

## 文档索引

- `docs/architecture/<文档>.md` -- 一句话说明
- `docs/conventions/<文档>.md` -- 一句话说明
- `docs/workflows/<文档>.md` -- 一句话说明
```

## docs/ 文档规范

### 文档模板

所有 docs/ 文件遵循统一模板（详见 [templates.md](templates.md)）：

```markdown
# 文档标题

> When to read: 一句话说明 Agent 什么场景下该读此文件

## Overview

2-3 句话概述。

## Key Concepts

- **概念A**: 解释
- **概念B**: 解释

## Patterns

具体代码模式、架构决策、示例代码。

## Gotchas

非直觉行为、常见陷阱、历史遗留问题。
```

### 分类指引

| 目录            | 放什么                                   | 示例                             |
| --------------- | ---------------------------------------- | -------------------------------- |
| `architecture/` | 系统分层、模块关系、技术选型、运行时机制 | 微服务架构、模块联邦、数据库设计 |
| `domains/`      | 业务实体、领域概念、数据流、接口协议     | 用户模型、订单流程、消息协议     |
| `conventions/`  | 命名规范、组件模式、样式规则、测试策略   | 代码风格、API 设计规范、错误处理 |
| `workflows/`    | 开发命令、构建流程、发布步骤、Git 工作流 | 本地开发、CI/CD、版本发布        |

### 文档质量标准

- 每个文件 **100-200 行**，超出则拆分
- 必须包含 **"When to read"** -- 帮助 Agent 判断相关性
- **Gotchas 是最高价值内容** -- 记录 Agent 从代码推断不出的坑
- 标题最多 **3 级**，需要 h4 时拆文件

## 格式规则

1. **用标题分区** -- 一个标题一个主题，最多 3 级
2. **命令用反引号** -- `npm run build` 一眼可识别为可执行指令
3. **每条规则附带理由** -- `禁止 force push -- 会覆盖共享历史`
4. **用列表不用段落** -- 列表逐项处理，段落中的指令容易被遗忘
5. **用标准节名** -- Commands, Architecture, Conventions, Boundaries
6. **层级要浅** -- 3 级标题封顶，需要 h4 时拆文件
7. **可立即执行** -- 每条指令通过"能否立即行动"测试

## 多应用/多模块项目

- 根 AGENTS.md 放**跨应用/跨模块**的全局规则
- 每个可独立开发的应用/模块有自己的 AGENTS.md
- 根 docs/ 放跨应用关注点，应用 docs/ 放应用专属内容
- 应用级文档必须**自足** -- 独立开发时不依赖根目录文档

## 维护策略

| 触发事件                 | 操作                                           |
| ------------------------ | ---------------------------------------------- |
| 架构变更                 | 更新 `docs/architecture/`                      |
| 操作流程变更             | 更新 `skills/` 或 `docs/workflows/`            |
| Agent 反复犯同一个错     | 写入 AGENTS.md（高频规则）或 docs/（详细背景） |
| Agent 已能正确处理的规则 | 从 AGENTS.md 删除，减少噪音                    |

### 定期审查

- **每月**：审查 AGENTS.md，精简冗余规则
- **每季度**：docs/ 跟随架构迭代
- **重点标记**：对关键规则使用 `IMPORTANT` / `CRITICAL`

## 执行清单

创建或重构项目 Agent 文档时，按以下顺序执行：

1. **分析项目** -- 了解技术栈、目录结构、构建工具、已有文档
2. **创建 README.md** -- 优先写清安装、初始化、构建、测试的最短上手路径
3. **创建 AGENTS.md** -- 根据项目实际情况填充必选节
4. **创建 CLAUDE.md** -- `@AGENTS.md` + Claude 专属指令
5. **规划 docs/** -- 根据项目复杂度决定需要哪些文档
6. **编写 docs/ 文档** -- 按模板填充，从 architecture/ 开始
7. **多应用处理** -- 为每个独立应用创建 AGENTS.md 和 docs/
8. **验证** -- README.md 是否能支持首次初始化；AGENTS.md <80 行；docs/ 文件 <200 行；所有文档有 "When to read"
